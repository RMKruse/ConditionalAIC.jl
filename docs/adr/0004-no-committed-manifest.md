# `Manifest.toml` is not committed; the exact `MixedModels` pin is the reproducibility guarantee

**Status:** accepted (2026-05-27). Supersedes the earlier "commit the top-level
`Manifest.toml`" rule.

## Context

`cAIC.jl` depends on `MixedModels.jl` **internals**, so the *exact* `MixedModels`
version must be fixed and bumped only as a deliberate, reviewed event. This is
enforced by an exact pin in `[compat]` — `MixedModels = "=5.5.1"` — in **both**
`Project.toml` and `test/Project.toml`.

The project additionally committed the top-level `Manifest.toml` to pin the whole
transitive dependency tree. That manifest broke CI: it was resolved on the
developer's local Julia (`1.12.6`), and the CI matrix runs `1.10`, `1.11`, and
`nightly`. On `1.10`/`1.11` the jobs failed at *instantiate*, before any test ran:

```
The active manifest file has dependencies that were resolved with a different
julia version (1.12.6). … Could not locate the source code for the OpenSSL_jll
package. Are you trying to use a manifest generated by a different version of Julia?
```

`OpenSSL_jll` is a bundled **stdlib JLL on Julia 1.12** but an ordinary registered
package on 1.10/1.11. A manifest resolved on one Julia version records stdlib JLLs
in a form the other versions cannot instantiate. This is general, not specific to
`OpenSSL_jll`: **a single committed `Manifest.toml` cannot serve a multi-Julia CI
matrix**, because the set and provenance of stdlib JLLs differ per Julia version.
This is exactly why the Julia community commits a `Manifest.toml` for *applications*
but not for *libraries* — and `cAIC.jl` is a library, bound for the General registry.

## Decision

**Do not commit any `Manifest.toml`** (top-level or sub-environment). `.gitignore`
ignores `**/Manifest.toml` with no exception. The reproducibility guarantee for the
quarantined `MixedModels` dependency is the **exact `=X.Y.Z` compat pin**, present in
both `Project.toml` and `test/Project.toml`. Bumping that pin remains the deliberate,
reviewed event; CI resolves the rest of the tree fresh against `[compat]` on each
supported Julia version.

## Why this is sufficient

- **The pin is the real guarantee.** The point of the manifest rule was that a
  `MixedModels` change is never incidental. The exact `[compat]` pin delivers exactly
  that — the version cannot move without editing `Project.toml`/`test/Project.toml`.
  The full transitive manifest added per-Julia-version reproducibility at the cost of
  being un-instantiable across the matrix.
- **Internals are still pinned.** The internal-access audit table in
  `src/mm_internals.jl` is keyed to the pinned `MixedModels` version, independent of
  whether a manifest is committed.
- **No more cross-version instantiate failures.** Each CI job resolves against
  `[compat]` for its own Julia, so stdlib-JLL provenance always matches the runner.

## Considered alternatives

- **Version-specific manifests** (`Manifest-v1.10.toml`, `Manifest-v1.11.toml`; Julia
  auto-selects the matching one). Honors a "committed, reproducible" goal per version,
  but multiplies maintenance — every manifest must be regenerated on each dependency
  or Julia bump, `nightly` still has none, and generation needs every matrix Julia
  installed locally. Rejected: maintenance burden out of proportion to the benefit for
  a library already pinned by exact `[compat]`.
- **Regenerate the single `Manifest.toml` on the lowest supported version (1.10).**
  One committed file, instantiable on 1.10; but 1.11/`nightly` still re-resolve forward
  (the "resolved with a different julia version" warning persists), so it is not the
  reproducible artifact a committed manifest is supposed to be. Rejected as a
  half-measure.

## Consequences

- `git rm --cached Manifest.toml`; the working-copy file remains, untracked, so local
  development on the developer's Julia is unaffected.
- A `MixedModels` bump is reviewed by editing the exact `[compat]` pin in both project
  files and walking the `src/mm_internals.jl` audit table — there is no manifest diff
  to review.
- Should byte-exact full-tree reproducibility ever be required (e.g. chasing a
  transitive-dependency regression), the version-specific-manifest alternative above
  can be revisited in a follow-up ADR.
